This repo contains a compiler, an OCaml VM, a JS code generator (emit-js), and an LLVM native backend for a new language called MiniML. It has two compilers, an OCaml reference implementation and the MiniML self-hosted version. PATH-B CUTOVER DONE (roadmap #16/#16b, see docs/freeze-point.md): self_host/ is now the SOURCE OF TRUTH for the compiler — to change the compiler, EDIT self_host/ DIRECTLY (the self_host/{main,driver,mml}.mml entries were always hand-maintained; now the whole tree is). `make translate-all` is NO LONGER part of the build — it is a deliberate RE-SYNC-FROM-REFERENCE tool that OVERWRITES the translated self_host/*.mml from the frozen lib/ reference, so do NOT run it during normal development (it clobbers direct edits). The production toolchain is the self-host native `mml` (run/build/check/fmt/get); rebuild its JS artifacts from the canonical self_host/ with `make self-host-compile-js` (js/compiler.json, parity) and `make self-host-compile-native-js` (js/compiler_native.js, playground + the OCaml-free bootstrap seed). The OCaml reference (lib/) is now FROZEN as: the executable spec (oracle, lib/oracle.ml), the bootstrap, and a differential cross-check. The differential gates (parity/ir-parity/translate) remain GREEN and serve as a frozen-reference regression net; they are retired only if/when a deliberate self_host/ change legitimately diverges from the frozen lib/. Self-host-direct verification (no reference diff): `make test-selfhost-native` (corpus through mmlc) + `make bootstrap-ocaml-free` (node-only build) + the playground stage. Both compilers share a common MiniML standard library in stdlib/ . You can rebuild the self-host compiler for the parity tests by running `make self-host-compile-js` (the parity runner uses js/compiler.json on the OCaml VM) and for the web playground by running `make self-host-compile-native-js` (js/compiler_native.js). You can find a lot of useful commands in the Makefile. There's also a web playground in the js/ folder; it runs exclusively on emit-js output (the JS bytecode VM was retired, roadmap #13c).

TESTING: `make check` is THE pre-merge gate — it runs every suite on every backend with both compilers (unit, translate, emit-js, playground, native, oracle, parity, fuzz, diff, ir-parity). The ir-parity stage (`make ir-parity-full`) diffs the LOWERED IR (`--emit-ir`) of both compilers over the whole corpus and fails on any NEW cross-compiler structural/type divergence (tolerating a small documented baseline of alpha-equivalent residuals in compiler_test/ir_parity_runner.ml) — it catches value-invisible faithfulness bugs value parity misses. Suites run in parallel (CHECK_JOBS=n, default 4; CHECK_JOBS=1 for sequential), each logging to /tmp/mml-check-logs/<suite>.log with a one-line PASS/FAIL + timing summary; failing suites' logs are dumped at the end. The fuzz stage (`make test-fuzz`) runs 50 fixed-seed generated programs differentially on oracle/vm/emit-js — a failure there is a real, reproducible semantic divergence (replay with `make fuzz COUNT=1 SEED=n`); the generator's avoidance ledger (diff_test/generator.ml comments) lists the constructs it skips because of tracked open bugs. Nothing merges to main unless `make check` is green. Don't run other CPU-heavy work (fuzzing, extra suites) concurrently with a gate run — it starves the gate. While iterating, use the individual targets: `make test-ocaml` first (fastest, OCaml VM), then the suite closest to what you changed (`make test-emit-js` for the JS codegen, `make test-playground` for the self-host-compiler-on-JS path that the web playground uses, `make test-native` for LLVM, `make test-oracle` for the reference interpreter, `make test-parity` for self-host bytecode — the slowest). test-emit-js/test-playground run tests concurrently (CROSS_TEST_JOBS=n to override, =1 for sequential); output stays in source order. When fixing compiler bugs make sure to add comprehensive test cases in the cross_test/ folder; per-backend skip directives (`--- skip-native:`, `--- skip-emit-js:`, `--- skip-playground:`) must reference a tracked bug. SEMANTICS: lib/oracle.ml is the executable spec for evaluation semantics (docs/semantics.md is its prose form); when backends disagree about behavior, the oracle is the authority — implement language features in the oracle first, then the backend lowerings. Documentation about the language, like syntax, control structures, type system, and more can be found in the docs/ folder. The research/ folder contains older information and can be ignored unless explicitly requested. Also at the end of implementation summaries include any oddities, quirks, or bugs you had to workaround so they can be logged for later. Note that if statements in MiniML use `do/else/end` instead of `then/else` like ocaml, see the syntax doc for more nuance. When running the cross_test suites its good to write them to a /tmp/ file so you can do greps or statsitics later. It's a slow test so it's more efficient to run it once and then look at the output.
always prefer principled and correct solutions over pragmatic or tactical fixes, even if its more work. this is a production quality code base.
