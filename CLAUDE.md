This repo contains a compiler and two VMs (one OCaml, and one Javascript) for a new language called MiniML. It has two compilers, an OCaml reference implementation and the MiniML self-hosted version. The self-hosted version is mostly mechanically translated from the OCaml reference compiler using the tools/ocaml_to_mml executable, and can be run with `make translate-all`. The self_host/main.mml file is manually updated. Both compilers share a common MiniML standard library in stdlib/ . You can rebuild the self-host-compile for the parity tests and web by running `make self-host-compile-js`, as the parity runner uses the JS compiler JSON. You can find a lot of useful commands in the Makefile. There's also a web playground and JS virtual machine in the js/ folder.

TESTING: `make check` is THE pre-merge gate — it runs every suite on every backend with both compilers (unit, js-suite, translate, js, emit-js, playground, native, parity), ordered fastest-first. Nothing merges to main unless `make check` is green. While iterating, use the individual targets: `make test-ocaml` first (fastest, OCaml VM), then the suite closest to what you changed (`make test-js` for the JS VM, `make test-emit-js` for the JS codegen, `make test-playground` for the self-host-compiler-on-JS path that the web playground uses, `make test-native` for LLVM, `make test-parity` for self-host bytecode — the slowest, run last). When fixing compiler bugs make sure to add comprehensive test cases in the cross_test/ folder; per-backend skip directives (`--- skip-native:`, `--- skip-emit-js:`, `--- skip-playground:`) must reference a tracked bug. Documentation about the language, like syntax, control structures, type system, and more can be found in the docs/ folder. The research/ folder contains older information and can be ignored unless explicitly requested. Also at the end of implementation summaries include any oddities, quirks, or bugs you had to workaround so they can be logged for later. Note that if statements in MiniML use `do/else/end` instead of `then/else` like ocaml, see the syntax doc for more nuance. When running the cross_test suites its good to write them to a /tmp/ file so you can do greps or statsitics later. It's a slow test so it's more efficient to run it once and then look at the output.
always prefer principled and correct solutions over pragmatic or tactical fixes, even if its more work. this is a production quality code base.
